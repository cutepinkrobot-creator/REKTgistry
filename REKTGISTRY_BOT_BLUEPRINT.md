# Twitter Bot Blueprint — RektGistry Edition
## Build instructions for a new Claude session

---

## What you are building

A Twitter engagement bot for **rektgistry.com** — same architecture as the QuanMed AI bot.

**What it does:**
- Runs on a schedule (3× daily) and searches Twitter for relevant tweets
- Sends suggested replies to a Telegram group for human approval
- Once approved, auto-posts the reply to Twitter from the main account
- Also suggests original tweets (short punchy + long informative variants)
- Supports an **English account** (manual posting — sends copy-ready text to Telegram)
- Photo generation via OpenAI gpt-image-1, overlaid with the brand logo
- All controlled via Telegram slash commands

## Core architecture — critical details

### Twitter client (twitter_browser.py)

**Do NOT use twikit or any third-party Twitter client.** Twitter's API requires `x-client-transaction-id` headers that are generated by browser-side JavaScript. No Python library can replicate this correctly as of 2025.

Instead: use **Playwright headless Chromium** with session cookies injected. The browser generates all required headers automatically.

**Critical cookie bug:** Twitter redirects to x.com. You must set cookies on BOTH `.twitter.com` AND `.x.com` domains, otherwise the browser treats itself as a guest:

```python
ctx.add_cookies([
    {"name": "auth_token", "value": auth_token, "domain": ".twitter.com", "path": "/"},
    {"name": "ct0",        "value": ct0,        "domain": ".twitter.com", "path": "/"},
    {"name": "auth_token", "value": auth_token, "domain": ".x.com",       "path": "/"},
    {"name": "ct0",        "value": ct0,        "domain": ".x.com",       "path": "/"},
])
```

**Critical event loop bug:** NEVER use `threading.Event.wait()` or `time.sleep()` inside Playwright sync API code. It blocks the event loop and response events never fire. Use `page.wait_for_timeout(500)` in a polling loop instead:

```python
# WRONG — blocks event loop, responses never arrive:
threading.Event().wait(timeout=8)

# CORRECT:
for _ in range(16):   # 16 × 500ms = 8s max
    if captured:
        break
    page.wait_for_timeout(500)
```

**Search approach:** Navigate to `https://twitter.com/search?q={query}&f=live&src=typed_query`, listen for `SearchTimeline` GraphQL responses, parse the JSON. Use `domcontentloaded` not `networkidle` (Twitter loads forever, networkidle always times out):

```python
page.goto(url, wait_until="domcontentloaded", timeout=10000)
```

**Performance:** Reuse the same browser page across multiple searches. First load takes ~20s (JS bundle), subsequent navigations take ~3s because the bundle is cached.

**Posting tweets:** Use `page.evaluate()` to run `fetch()` from inside the browser context. This way the browser's own CSRF token and transaction ID are used automatically:

```python
result = page.evaluate(f"""async () => {{
    const r = await fetch("https://twitter.com/i/api/graphql/c50A_puUoQGK_4SXseYz3A/CreateTweet", {{
        method: "POST",
        headers: {{
            "Authorization": "Bearer AAAAAAAAAAAAAAAAAAAAANRILgAAAAAAnNwIzUejRCOuH5E6I8xnZz4puTs%3D1Zv7ttfk8LF81IUq16cHjhLTvJu4FA33AGWWjCpTnA",
            "Content-Type": "application/json",
            "x-csrf-token": "{ct0}",
            "X-Twitter-Auth-Type": "OAuth2Session",
            "X-Twitter-Active-User": "yes",
        }},
        credentials: "include",
        body: {repr(payload)},
    }});
    const data = await r.json();
    return {{status: r.status, data: data}};
}}""")
```

The CreateTweet query ID `c50A_puUoQGK_4SXseYz3A` — verify this is still current by logging into Twitter in a browser, opening DevTools Network tab, posting a tweet, and checking the GraphQL request URL.

### Telegram approval flow (telegram_client.py)

**Pending state** is stored in `pending.json`. Each item has a `status` field that moves through:

```
awaiting_text
  → (approve) → text_approved         [for originals/english]
  → (approve) → ready                  [for Chinese replies — skip photo]
  → (reject)  → rejected

text_approved
  → (process_text_approvals) → awaiting_photo_decision

awaiting_photo_decision
  → (want_photo)  → wants_photo
  → (no_photo)    → ready

wants_photo
  → (process_text_approvals) → generating_photo → awaiting_photo

awaiting_photo
  → (approve_photo) → ready
  → (reuse_photo)   → ready
  → (skip_photo)    → ready
```

**Critical deadlock fix:** `on_ready` must be called OUTSIDE the `_lock` mutex. Calling it inside causes deadlock because `on_item_ready` calls `tg._update_pending()` which also tries to acquire `_lock`. Use a `post_action` tuple pattern:

```python
post_action = None

with _lock:
    # ... mutate state, save ...
    post_action = ("ready", dict(item))   # save what to do

# Fire AFTER releasing lock
if post_action:
    kind, saved_item = post_action
    if kind == "ready":
        on_ready(item_id, saved_item)
```

**Photo decision flow (important UX):** Do NOT auto-generate the image on text approval. First ask "Add a photo? Yes/No". Only generate if user says Yes. Otherwise the expensive image generation runs even for posts the user immediately skips photos for, and the "Reuse Previous" button is useless because a new image was already just generated.

**409 conflict:** If you also run the bot on a server (EC2 etc.), both instances will fight for the Telegram polling slot. The local instance force-claims it via `setWebhook`/`deleteWebhook` on startup. The EC2 bot will keep reclaiming every few seconds. Solution: only run one instance at a time, or stop the server bot when running locally.

**Item types:**
- `reply` — Chinese/main account reply, auto-posts on approval, no photo
- `original` — Chinese/main account original tweet, photo decision step, then auto-posts
- `english_reply` — English reply, photo decision step, then sends copy-ready message (manual)
- `english_original` — English original tweet, photo decision step, then sends copy-ready message (manual)

### Content generator (content_generator.py)

**KEYWORDS dict** — flat list of topic keywords used for relevance scoring. The `score_tweet_relevance()` function counts keyword matches and returns `matches / 5.0` (capped at 1.0). Set `MIN_RELEVANCE_SCORE = 0.2` (1 keyword match passes).

**Important:** Chinese keywords must be short atomic tokens that appear as substrings in natural sentences. Multi-word Chinese phrases like `"量子计算医疗"` won't match `"量子计算在医疗"` because the particle `在` breaks the match. Add short versions alongside the long ones.

**Reply generation:** The reply prompt must:
- Open by reacting to the content (not restating it)
- Mention `@YOURACCOUNT` in the middle, not at the end
- Sound like a peer/insider, not an ad
- End with one emoji

**English originals:** Two variants:
- Short/punchy: 200–260 chars, contrarian take, punchy
- Long/informative: 500–700 chars, detailed prose, explains mechanism/data, educational

### Bot logic (bot.py)

**Candidate filtering:**
```python
MIN_RELEVANCE_SCORE = 0.2   # 1 keyword match
MIN_FOLLOWERS = 1000         # ignore tiny accounts
MAX_SUGGESTIONS_PER_RUN = 1  # Chinese replies: 1 at a time
MAX_EN_SUGGESTIONS_PER_RUN = 5  # English replies: 5 at a time
MAX_PENDING_BEFORE_PAUSE = 3  # don't flood if user hasn't reviewed yet
```

**Always filter out reply tweets from candidates** (tweets that are themselves replies to others):
```python
if t.get("in_reply_to_user_id") or t.get("inReplyToUserId"):
    continue
```

**Random reply delay:** When auto-posting approved replies, add a random 30–180s delay so batch approvals don't post back-to-back (looks like a bot):
```python
delay = random.randint(30, 180)
threading.Timer(delay, _do_post_item, args=(item_id, item)).start()
```

**Caching:** Save candidate lists to JSON. `/next` pages through without re-searching. Separate cache files for Chinese (`candidates_cache.json`) and English (`en_candidates_cache.json`).

**process_text_approvals()** runs every 15s in a background thread. Handles:
1. `text_approved` → send "Add a photo?" question
2. `wants_photo` → generate image, send photo suggestion

### Scheduler (run.py)

Run with `python run.py --loop` for continuous operation. Without `--loop` it runs one scan then keeps the Telegram listener alive.

**Register commands with Telegram on startup** so they appear in the menu:
```python
tg._post("setMyCommands", commands=[
    {"command": "run",              "description": "..."},
    ...
])
```

**Never run multiple instances.** Use `ps aux | grep run.py` to check. Kill all before restarting.

## Gotchas and known issues (learn from these)

1. **Python 3.10+ required** if you use twikit — it uses `X | Y` union type syntax. Skip twikit entirely, use the Playwright approach above.

2. **Multiple bot instances** — if you restart without killing the old process, you'll have N instances all fighting for the Telegram polling slot and all logging to the same file. Always `ps aux | grep run.py` and kill all before restarting.

3. **`.env` not auto-loaded** — only `run.py` calls `load_dotenv()`. If you run ad-hoc test scripts, the browser will have empty auth cookies and return 0 tweets. Always load dotenv at the top of any test script:
   ```python
   from dotenv import load_dotenv
   load_dotenv(Path(".") / ".env", override=True)
   ```

4. **Score filter too strict** — if you find 0 candidates, check the filter breakdown. The score filter is the most common culprit. `MIN_RELEVANCE_SCORE = 0.2` (1 keyword match) is the right default — searches are already targeted so any keyword match is meaningful.

5. **Tweet character limits** — `generate_reply()` for the main account should target 80–120 chars (Chinese is compact). English replies: 180–240 chars. Long English: 500–700 chars (user trims manually). Never hard-cap with `text[:267] + "..."` — it truncates mid-sentence. Trust the prompt instead.

6. **Telegram 429 rate limit on setWebhook** — the `_force_claim_polling()` function hits Telegram's webhook API. If called too frequently (every few seconds during a 409 fight), Telegram rate-limits it with 429. This is logged as WARNING and is harmless — the bot keeps retrying.

7. **Photo "Reuse Previous" was broken** — because `process_text_approvals` was generating a new image before asking the user. The user sees a new image alongside "Reuse Previous" — but there's nothing obvious to reuse. Fix: ask first ("Add a photo?"), then generate only if yes.

8. **on_ready deadlock** — see Critical deadlock fix above. This is a subtle bug that can cause the bot to freeze silently after approving a suggestion.

9. **run.py exits after one scan** — without `--loop` flag, the main thread exits after the scan. The `while True: time.sleep(1)` keeps it alive for Telegram, but scheduled scans don't run. Always use `python run.py --loop` in production.

## How to get Twitter session cookies (step by step)

1. Open Chrome, go to x.com, log in to the account the bot will post from
2. Open DevTools (F12) → Application tab → Storage → Cookies → https://x.com
3. Find `auth_token` — copy its Value (40-char hex string)
4. Find `ct0` — copy its Value (long hex string, ~160 chars)
5. Put both in `.env`
6. **Do not log out** of this account on this browser — logging out invalidates the cookies
7. If the bot starts getting 403/401 errors, the cookies expired — repeat this process
